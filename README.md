# Overbrew

Overbrew adds brewable potions for normally unavailable effects, expanded duration and amplifier tiers for custom and vanilla potions, and particle-free variants.

Every configured potion form, variant grid, conversion, transformation, and global-modifier chain is generated from `src/generator/config.json`.

## Using the generator

Install the development dependencies:

```bash
bun install
```

Generate with the default config:

```bash
bun run generate
```

Generate with another config:

```bash
bun src/generator/index.ts path/to/config.json
```

Config-relative paths are resolved from the selected config file.

The generator clears and recreates the custom brewing recipe directory, `minecraft:recipe/brewing`, its generated item-tag root, and the vanilla tipped-arrow recipe. When enabled, it also regenerates the configured advancement subtree and potion-table file. Other datapack files are preserved. Before finishing, it validates potion-state matchers, generated recipes, tag references and overlaps, ambiguous brewing routes, and generation counts.

## Config layout

```text
generator    Target, namespaces, matching, tags, output, and manifest settings
forms        Potion item forms
bases        Base-potion brews
modifiers    Reusable brewing modifiers
custom       Overbrew effects and transformations
vanilla      Vanilla potion states and transformations
```

All generator-only metadata and behavior, including `target` and `namespaces`, lives under `generator`.

The shipped default configuration is available at `src/generator/config.json`.

## Generator settings

### Target and namespaces

```json
{
  "target": {
    "description": [
      {
        "text": "Expands brewing with missing potions, new duration and strength tiers, particle-free variants, and custom fuels.",
        "color": "gold"
      }
    ],
    "min_format": 110,
    "max_format": 111,
    "overlays": [
      {
        "directory": "overlay_111",
        "min_format": 111,
        "max_format": 111
      }
    ]
  },
  "namespaces": {
    "custom": "overbrew",
    "vanilla": "minecraft"
  }
}
```

- `target.description` is written directly to `pack.description` and accepts plain text or text components.
- `target.min_format` and `target.max_format` define the pack's supported integer data-pack formats.
- `target.overlays` writes format-gated overlay entries using the same integer range fields.
- `namespaces.custom` controls Overbrew recipe, advancement, and item-tag locations.
- `namespaces.vanilla` controls vanilla resources, recipe types, default namespaces, and reconstructed recipes.

Use complete resource locations in the rest of the config. Unnamespaced values use the configured vanilla namespace.

### Variant generation

```json
{
  "variant_generation": {
    "axes": {
      "duration": ["base", "long", "prolonged"],
      "amplifier": ["base", "strong", "potent"]
    },
    "profiles": {
      "standard": {
        "durations": 2,
        "amplifiers": 2,
        "mix": true,
        "states": {
          "base": { "duration": 6, "amplifier": 0 },
          "long": { "duration": 16, "amplifier": 0 },
          "strong": { "duration": 3, "amplifier": 1 }
        }
      },
      "duration_only": {
        "extends": "standard",
        "amplifiers": 0
      },
      "instant": {
        "durations": 0,
        "amplifiers": 2,
        "mix": false,
        "states": {
          "base": { "duration": 1, "amplifier": 0 },
          "strong": { "duration": 1, "amplifier": 1 }
        }
      }
    },
    "rounding": "nearest"
  }
}
```

The two axis arrays define the names and order of duration and amplifier tiers. Both must begin with `base`. Mixed names are generated by joining non-base names, producing states such as `long_strong`, `long_potent`, and `prolonged_potent`.

The shipped config uses the internal duration ID `long` for the player-facing **Extended** tier. Modifier IDs do not need to match axis IDs: the `extended` modifier moves a potion from `base` to `long`.

A profile may define `durations`, `amplifiers`, `mix`, reusable `states`, and optional `scaling`. Potion families inherit those fields through `variants.profile` and only override what differs. Profiles may inherit another profile through `extends`; state entries and nested scaling maps are merged instead of replacing the complete parent profile.

Profile states describe a reusable progression rather than requiring the potion's real durations. For example, the `standard` profile expresses the same relationship as Regular 3:00, Extended 8:00, and Strong II 1:30 using the smaller equivalent values `6`, `16`, and `3`. A family supplies its real `base` state, and the generator applies the profile's relative duration and amplifier changes.

Profiles can also define exact multi-effect patterns. Turtle Master, for example, stores `base`, `long`, and `strong` arrays once in its profile instead of repeating per-effect amplifier rules in every potion family.

When a calculation is clearer as an operation, `scaling` remains available. A multiplier accepts either a number or an exact fraction:

```json
{
  "scaling": {
    "duration": {
      "multiplier": 2
    },
    "amplifier": {
      "duration_multiplier": {
        "numerator": 1,
        "denominator": 2
      }
    }
  }
}
```

A number is shorthand for the same value over `1`. Fractions avoid approximate decimal values such as `2.66666667`.

Calculation follows these rules:

1. The family effect template and local `states.base` create the real starting state. A profile's `base` state is used only when the family does not supply one.
2. Local state values override profile-derived or scaled values for that state.
3. A profile's named state is applied relative to its `base` state.
4. Missing later duration tiers follow `scaling.duration.progression`. `linear` repeats the Regular-to-Extended increase, so 3:00 -> 8:00 becomes 13:00; `multiplicative` repeats the ratio instead.
5. Missing amplifier tiers remain sequential: Potent upgrades Strong using the first amplifier step and duration relationship.
6. Explicit `scaling` is used when no named profile state supplies that tier.
7. Per-effect state arrays or scaling maps support multi-effect potions.
8. Mixed states combine the independently resolved duration and amplifier axes.
9. `rounding` is applied after every duration calculation and accepts `nearest`, `floor`, or `ceil`.

A family may omit `profile`, but it must then provide `states.base` and enough local scaling or explicit first-tier states to resolve every enabled axis. When a first-tier state is explicit and no scaling rule is supplied, the generator derives the repeated tier relationship from `base` to that state.

### Vanilla recipes

```json
{
  "include_vanilla_recipes": true
}
```

Before generation, the complete `data/<vanilla namespace>/recipe/brewing` directory is removed so recipes deleted or renamed in the config cannot leave stale files behind.

When enabled, the generator reconstructs vanilla brewing recipes under the configured vanilla namespace and routes their reagents through the item-tag system. A base-potion input receives `effects.size: 0` only when the active configuration can assign that exact potion type to a custom-effect state. This keeps ordinary vanilla predicates unchanged unless the guard is needed.

When disabled, the cleared vanilla brewing directory remains empty while Overbrew recipes and configured global-modifier chains are still generated. Active `potion_type` strategies are skipped because their base recipes cannot be guarded safely; nested duration and effect fallbacks remain available.

### Component-preserving tipped arrows

```json
{
  "preserve_imbued_components": true
}
```

This setting defaults to `true`. It replaces the vanilla `tipped_arrow.json` recipe with a `crafting_transmute` recipe that combines one configured lingering-potion item with eight arrows and produces eight tipped arrows.

The transmutation copies the lingering potion's components to the arrows, allowing custom potion contents, colors, lore, and other compatible components to survive crafting. The generated output removes `minecraft:custom_name` so the result uses the tipped-arrow name instead of the potion name.

The replacement is written to:

```text
data/<vanilla namespace>/recipe/tipped_arrow.json
```

The configured `lingering` form must exist while this setting is enabled. Set the option to `false` to omit the replacement recipe.

### Potion matching and conflict detection

```json
{
  "potion_matching": {
    "detect_conflicts": true,
    "check_tag_overlaps": true,
    "effect_fields": {
      "duration": true,
      "amplifier": true,
      "ambient": true,
      "visible": true
    }
  }
}
```

Minecraft brewing ingredients can inspect only the potion item and these fields of the `minecraft:potion_contents` predicate:

- `potions`: one or more registered potion types or a potion tag. The generator uses only this plural field.
- `effects`: effect IDs, collection size/count rules, and the matchable properties below.

| Predicate field | Output field     | Behavior                                                                                                                                                         |
| --------------- | ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `duration`      | `duration`       | Matches the effect duration in ticks.                                                                                                                            |
| `amplifier`     | `amplifier`      | Matches the zero-based effect level.                                                                                                                             |
| `ambient`       | `ambient`        | Matches whether the effect is marked as ambient. Ambient effects use subtler beacon-style particles, but particles are still present when visibility is enabled. |
| `visible`       | `show_particles` | Matches whether effect particles are shown. This does not control the inventory/HUD icon.                                                                        |

All four fields are enabled by default so generated predicates are as specific as Minecraft allows.

`show_icon` and `hidden_effect` can be written to `custom_effects`, including recursively nested hidden effects, but the brewing predicate cannot inspect them. It also cannot inspect `custom_color`, `minecraft:custom_name`, lore, custom data, or other item components. These values therefore cannot resolve an ambiguous brewing recipe.

When `detect_conflicts` is enabled, generation stops for:

- configured potion states with the same matchable predicate;
- recipes that can accept the same potion item, potion contents, and reagent but produce different outputs;
- modifiers that can accept their own output and repeat indefinitely.

Errors identify the conflicting states or recipes and suggest the relevant setting or data change.

#### Item-tag overlap checks

`check_tag_overlaps` defaults to `true`. The validator recursively expands generated Overbrew item tags before comparing recipes, so two differently named tags still conflict when they contain the same item.

Set it to `false` only when overlapping generated tags are intentional. External tags that the generator does not own are handled conservatively because their complete contents are unavailable during generation.

### Automatic modifier-output adjustment

Component-only global modifiers can create potion outputs that differ in lore, custom data, or other components that brewing predicates cannot inspect. Without a matchable distinction, the same modifier could accept its own output or collide with another recipe.

Automatic disambiguation applies to every global modifier by default. The shipped configuration preserves compatibility with existing Overbrew potions by using only duration adjustment when a conflict actually occurs:

```json
{
  "auto_adjust_effects": {
    "enabled": true,
    "strategies": [
      {
        "type": "duration",
        "priority": 10,
        "effects": "all",
        "prefer": "increase",
        "algorithm": "balanced",
        "order": "effect_first",
        "max_delta": 20,
        "max_total_delta": 20
      }
    ]
  }
}
```

The generator changes only a conflicting modifier output. It never edits the original potion state or the other potion involved in the conflict. Outputs that are already distinguishable remain unchanged. The strategy list comes from `config.json`; the generator does not inject a second hardcoded default list.

#### Potion types and existing-potion compatibility

Potion types are fixed registry entries, not datapack-defined resources. An Uncraftable Potion has no `potion` field; `minecraft:empty` is not a valid potion ID. Omitting `potions` from a brewing predicate means "do not check the potion type", not "require no potion type".

`default_potion_type` therefore defaults to `null`. Generated custom-effect potions omit the field, matching older Overbrew versions. Existing potions remain valid recipe inputs as long as their matchable effect data has not changed.

Changing only a conflicting output to another potion type does not work when its parent input is untyped: the parent's effects-only predicate would also accept the typed output. Potion-type disambiguation therefore requires a non-null `default_potion_type`, which intentionally starts a typed-potion migration. Older untyped potions remain usable, but recipes requiring the configured type will not accept them.

Potion-type identities are configured entirely through `config.json`. Each candidate must resolve to an effectless potion modeled by `bases` or by an empty entry under `vanilla.potions`; the generator contains no separate hardcoded identity list. With the shipped base model, a migration can use Awkward, Mundane, and Thick:

```json
{
  "potion_matching": {
    "default_potion_type": "minecraft:awkward",
    "auto_adjust_effects": {
      "strategies": [
        {
          "type": "potion_type",
          "priority": 10,
          "potions": [
            "minecraft:mundane",
            "minecraft:thick",
            "minecraft:awkward"
          ],
          "include_original": true,
          "strategies": [
            {
              "type": "duration",
              "priority": 10,
              "effects": "all",
              "prefer": "increase",
              "algorithm": "balanced",
              "order": "effect_first",
              "max_delta": 20,
              "max_total_delta": 20
            }
          ]
        }
      ]
    }
  }
}
```

This mode requires `include_vanilla_recipes: true`. Only the potion types reachable from the active default and strategy tree receive `effects.size: 0`; unrelated Water, Awkward, Mundane, Thick, or other base predicates are left untouched.

#### Strategy priority and modifier overrides

Strategies are tried from the lowest `priority` to the highest. Equal priorities retain declaration order. A strategy stops as soon as it produces a unique matchable state.

The shared strategy list applies to every global modifier by default. `modifiers` is only for exceptions:

```json
{
  "auto_adjust_effects": {
    "enabled": true,
    "modifiers": {
      "example_modifier": false,
      "special_modifier": {
        "strategies": [
          {
            "type": "duration",
            "max_delta": 5
          }
        ]
      }
    }
  }
}
```

A boolean disables or enables adjustment for one modifier. An object may override `enabled` or replace the shared strategy list.

#### Effect selection

The `effects` field selects which existing effects a strategy may change:

| Value     | Behavior                                               |
| --------- | ------------------------------------------------------ |
| `"all"`   | Tries every effect in list order. This is the default. |
| `"first"` | Changes only the first effect.                         |
| `[0, 2]`  | Changes only the listed zero-based effect indexes.     |

#### Numeric search algorithms

Numeric strategies always try the smallest total absolute adjustment first.

| Option                | Values                            | Behavior                                                                                                          |
| --------------------- | --------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| `algorithm`           | `minimal`, `balanced`, `single`   | Orders one-effect and multi-effect candidates. The shipped config uses `balanced`.                                |
| `order`               | `effect_first`, `direction_first` | Chooses whether effect order or preferred direction wins equal-cost ties. The shipped config uses `effect_first`. |
| `prefer`              | `increase`, `decrease`            | Chooses the preferred sign for equal-cost candidates. The shipped config uses `increase`.                         |
| `max_delta`           | Non-negative integer              | Limits the absolute adjustment applied to one effect.                                                             |
| `max_total_delta`     | Non-negative integer              | Limits the sum of absolute adjustments across all changed effects. Defaults to `max_delta`.                       |
| `max_changed_effects` | Non-negative integer              | Limits how many selected effects one candidate may change. Defaults to all selected effects.                      |
| `max_candidates`      | Positive integer                  | Advanced safety override. Omit it to use the internal limit of `10000`.                                           |

The shipped config uses `balanced`. After minimizing total change, it minimizes the largest individual change, so a total adjustment of three prefers combinations such as `+1 +2` before `+3 0`.

`minimal` instead prefers changing fewer effects, so it tries `+3 0` before `+1 +2` when both have the same total adjustment.

`single` never changes more than one effect.

The explicit `effects` order and `prefer` direction are respected as tie-breakers after the selected algorithm's cost rules.

#### Available strategies

| Type          | Behavior                                                                                                                |
| ------------- | ----------------------------------------------------------------------------------------------------------------------- |
| `potion_type` | Advanced migration strategy that cycles through configured effectless potion types and can interleave nested fallbacks. |
| `duration`    | Changes existing effect durations by the nearest number of ticks.                                                       |
| `amplifier`   | Changes effect levels. This has an obvious gameplay impact and should be used carefully.                                |
| `ambient`     | Sets or toggles the selected effects' ambient flag.                                                                     |
| `visible`     | Sets or toggles `show_particles` for the selected effects.                                                              |
| `add_effect`  | Appends a configured custom effect and may define nested fallback strategies.                                           |

An `add_effect` strategy tries the configured effect exactly as written first. If it still conflicts, explicit nested strategies and compatible inherited parent strategies are merged by `priority`. A nested strategy without `effects` targets only the newly added effect. Parent `add_effect` entries are not inherited, preventing recursive growth.

```json
{
  "type": "add_effect",
  "priority": 30,
  "effect": {
    "id": "example:marker",
    "duration": 1,
    "amplifier": 0,
    "ambient": true,
    "show_particles": false,
    "show_icon": false
  },
  "inherit_parent_strategies": true,
  "strategies": [
    {
      "type": "duration",
      "priority": 10,
      "prefer": "increase",
      "max_delta": 5
    }
  ]
}
```

Nested `add_effect` strategies are supported up to 16 levels, though simpler strategy trees are easier to maintain.

### Potion tables

```json
{
  "tables": {
    "enabled": true,
    "output": "../../POTION_TABLES.md",
    "sources": ["custom", "vanilla"],
    "include_ingredients": true,
    "underline_existing": true,
    "titles": {
      "custom": "Overbrew Potions",
      "vanilla": "Vanilla Potions"
    },
    "ingredient_names": {
      "minecraft:cod": "Raw Cod"
    }
  }
}
```

The generator writes one compact Markdown table per selected source. Regular, Extended, and Prolonged duration tiers are stacked inside each amplifier cell, with a line break between them. Durations are rounded to whole seconds for player-facing documentation.

Potion variants that already exist in vanilla are underlined individually. Generator-added vanilla variants, including Prolonged, Potent, mixed states, and Strong Weakness, remain unformatted. Multi-effect potions such as Turtle Master show every effect in the same cell.

- `output` is resolved relative to `config.json`.
- `sources` controls which tables are written and their order.
- `include_ingredients` enables the ingredient column.
- `underline_existing` highlights variants already available in vanilla.
- `titles` overrides section headings.
- `ingredient_names` supplies player-facing names when an item ID does not match its display name.

### Potion advancements

```json
{
  "advancements": {
    "enabled": true,
    "overlay": "overlay_111",
    "root": "overbrew",
    "entries": {
      "root": {
        "parent": null,
        "display": {
          "title": "Overbrew",
          "description": "Expands brewing with missing potions, new duration and strength tiers, and a resin modifier that removes particles.",
          "icon": {
            "id": "minecraft:potion",
            "components": {
              "minecraft:potion_contents": { "custom_color": 5578058 }
            }
          },
          "background": "minecraft:block/resin_block",
          "show_toast": false,
          "announce_to_chat": false
        },
        "match": [
          {
            "source": "custom",
            "effect_fields": {
              "duration": false,
              "amplifier": false,
              "ambient": false,
              "visible": false
            }
          },
          {
            "source": "vanilla",
            "registered": false,
            "effect_fields": { "visible": false }
          },
          {
            "source": "vanilla",
            "registered": true,
            "show_particles": false
          }
        ],
        "requirements": "any"
      },
      "overbrewer": {
        "parent": "root",
        "display": {
          "title": "Overbrewer",
          "description": "Brew any potion introduced or modified by Overbrew.",
          "icon": "minecraft:brewing_stand"
        },
        "match": [
          { "source": "custom" },
          { "source": "vanilla", "registered": false },
          { "source": "vanilla", "registered": true, "show_particles": false }
        ],
        "requirements": "any"
      }
    }
  }
}
```

The generator builds exact `minecraft:brewed_potion` criteria from the resolved potion graph. The configured tab root and its gameplay advancements are generated under the selected advancement root. Custom particle-free potions and supported vanilla particle-free variants can independently satisfy both the root and Invisible Chemistry checks.

Each entry supports:

- `parent`: a path relative to `root`, a complete advancement ID, or `null` for a tab root.
- `display`: title, description, icon, frame, background, toast, chat, and hidden settings. Modern backgrounds use a texture ID such as `minecraft:gui/advancements/backgrounds/stone`, without `textures/` or `.png`.
- `match`: one filter object or an array of filter objects interpreted as OR alternatives.
- `criteria`: explicit advancement criteria for non-potion triggers. It cannot be combined with `match`.
- `match.source`: `custom`, `vanilla`, or both.
- `match.family` and `match.variant`: one ID or a list of accepted IDs.
- `match.duration` and `match.amplifier`: configured axis names such as `prolonged` and `potent`.
- `match.registered`: selects variants already available in vanilla or generator-owned variants.
- `match.show_particles`: `true` or `false`; omitting it accepts either particle state.
- `match.effect_fields`: per-advancement overrides for duration, amplifier, ambient, and visibility matching. This can substantially reduce broad advancement files without changing recipe predicates.
- `requirements`: `any` for one matching state or `all` for every generated criterion.

The directory selected by `root` is cleared and regenerated inside the selected overlay. Other advancement folders in that overlay, including recipe-unlock advancements, are preserved.

### Format-111 brewing fuels

The two fuels are stored directly under `src/datapack/overlay_111`; the generator does not create or modify them.

- **Potent Blaze Powder:** 10 uses at 2x brewing speed.
- **Volatile Blaze Powder:** independently rolls 4, 10, or 24 uses and 0.5x, 2x, or 4x brewing speed.

Both use `minecraft:brewing_fuel` with generic `minecraft:number_provider` resources. Environmental speed dispatch was removed because the Snapshot 4 implementation did not apply the selected speed consistently even though the location predicate matched.

Obtaining Blaze Powder unlocks both custom fuel recipes through `overbrew:recipes/fuel/blaze_powder`.

### Output

```json
{
  "output": {
    "root": "../datapack/data",
    "manifest": "../../../dist/generator-manifest.json"
  }
}
```

- `root` points to the datapack's `data` folder and defaults to `./data` relative to the selected config.
- `manifest` is optional and writes the same generation summary printed to standard output.

The summary includes the target metadata, resolved namespace paths, matching settings, per-stage generation counts, validated recipe total, and generated item-tag total. Both paths are relative to the config file.

## Brew maps

Bases, custom effects, and directly brewed vanilla potions use the same input-to-reagent map:

```json
{
  "brew": {
    "awkward": {
      "items": ["minecraft:golden_apple"]
    }
  }
}
```

This means:

```text
Awkward Potion + Golden Apple -> configured output
```

Several input potions may create the same output:

```json
{
  "brew": {
    "awkward": {
      "items": ["minecraft:golden_apple"]
    },
    "thick": {
      "items": ["minecraft:enchanted_golden_apple"]
    }
  }
}
```

Each input receives a separate recipe for every configured potion form.

## Reagents

A reagent defines the accepted item values and optional tag behavior:

```json
{
  "items": ["minecraft:redstone"]
}
```

Several items can share one reagent tag:

```json
{
  "items": ["minecraft:redstone", "minecraft:redstone_block"]
}
```

A brewing recipe can reference only one item or one item tag. A multi-item reagent therefore needs an enabled tag or another generated tag that wraps it.

### Reagent-owned tags

Reagent values are defined beside the base, effect, or modifier that owns them. Their default singular paths are:

```text
base/<id>
effect/<id>
modifier/<id>
```

With the default root, examples are:

```text
#overbrew:brewing/base/awkward
#overbrew:brewing/effect/absorption
#overbrew:brewing/modifier/extended
```

The optional `tag` key controls the reagent's own tag.

```json
{
  "items": ["minecraft:redstone"],
  "tag": true
}
```

```json
{
  "items": ["minecraft:redstone"],
  "tag": false
}
```

```json
{
  "items": ["minecraft:redstone"],
  "tag": "modifier/duration"
}
```

```json
{
  "items": ["minecraft:redstone"],
  "tag": {
    "enabled": true,
    "path": "modifier/duration",
    "replace": false
  }
}
```

When `tag` is omitted, the default tag is enabled when its category appears in `generator.item_tags.auto_generate`.

When a tag is disabled, references to it are flattened to its configured items or nearest enabled parent tag. The final recipe reference must still resolve to exactly one item or tag.

## Item-tag generation

```json
{
  "item_tags": {
    "enabled": true,
    "root": "brewing",
    "auto_generate": ["base", "effect", "modifier"],
    "rules": []
  }
}
```

### Settings

| Key             | Behavior                                      |
| --------------- | --------------------------------------------- |
| `enabled`       | Enables generated reagent tags globally.      |
| `root`          | Sets the generated path below `tags/item`.    |
| `auto_generate` | Adds ordered automatic tag layers.            |
| `rules`         | Adds a final tag to matching recipe contexts. |

If `enabled` is `false`, every recipe must resolve to one direct item.

### Automatic families

Only these exact singular values are accepted:

| Family       | Example                                              | Applies to                  |
| ------------ | ---------------------------------------------------- | --------------------------- |
| `base`       | `base/awkward`                                       | Base-potion brews           |
| `effect`     | `effect/absorption`                                  | Direct effect brews         |
| `modifier`   | `modifier/extended`                                  | Modifier reagents           |
| `variant`    | `variant/absorption_to_long_absorption`              | Same-effect variant changes |
| `convert`    | `convert/leaping_to_slowness`                        | Cross-effect changes        |
| `conversion` | `conversion/regular_to_splash`                       | Potion-form changes         |
| `recipe`     | `recipe/custom/absorption/modifier/extended/regular` | Individual recipe contexts  |

Order matters. Each applicable family wraps the previously resolved reagent reference. Only tags referenced by generated recipes are written.

```json
{
  "auto_generate": ["modifier", "variant", "recipe"]
}
```

### Rules

Rules are checked in declaration order. The first matching rule wins.

```json
{
  "rules": [
    {
      "path": "modifier/extended/swiftness",
      "values": ["$auto", "minecraft:redstone_block"],
      "match": {
        "effect": "swiftness",
        "modifier": "extended"
      }
    }
  ]
}
```

Use `matches` for alternative match objects:

```json
{
  "path": "modifier/speed_duration",
  "values": ["$default"],
  "matches": [
    {
      "effect": "swiftness",
      "modifier": "extended"
    },
    {
      "effect": "haste",
      "modifier": "extended"
    }
  ]
}
```

Rule values:

| Value               | Meaning                                                  |
| ------------------- | -------------------------------------------------------- |
| `$default`          | The reagent's base, effect, or modifier tag              |
| `$auto`             | The most specific automatic tag resolved before the rule |
| `minecraft:item_id` | A direct item                                            |
| `#namespace:path`   | An external item tag                                     |
| `#local/path`       | A generated tag below the configured root                |

Available match keys:

```text
source
input
output
input_form
output_form
effect
input_effect
output_effect
variant
modifier
category
recipe
operation
recipe_key
input_particles
output_particles
```

A match value may be a string, number, boolean, or array. Arrays use OR behavior.

A rule path matching `convert/<input>_to_<output>` may omit its matcher. The generator then infers configured input/output variants where applicable.

## Base potions

```json
{
  "bases": {
    "awkward": {
      "brew": {
        "water": {
          "items": ["minecraft:nether_wart"]
        }
      }
    }
  }
}
```

The object key is the output potion ID and default base tag ID.

Configured bases are also included in the automatically derived list of vanilla potions accepted by container conversions.

## Modifiers

Every modifier defines a reusable reagent and may define one behavior type.

### Variant-axis transformation

```json
{
  "extended": {
    "reagent": {
      "items": ["minecraft:redstone"]
    },
    "variant_axis": {
      "axis": "duration",
      "to": "long",
      "from": ["base"]
    }
  },
  "prolonged": {
    "reagent": {
      "items": ["minecraft:redstone_block"]
    },
    "variant_axis": {
      "axis": "duration",
      "to": "prolonged",
      "from": ["long"]
    }
  }
}
```

`variant_axis` changes one configured axis while preserving the other. Duration modifiers therefore work across Regular, Strong, and Potent states, while amplifier modifiers work across Regular, Extended, and Prolonged states.

The shipped modifier set is:

| Modifier    | Reagent         | Axis target             |
| ----------- | --------------- | ----------------------- |
| `extended`  | Redstone Dust   | `duration -> long`      |
| `prolonged` | Redstone Block  | `duration -> prolonged` |
| `strong`    | Glowstone Dust  | `amplifier -> strong`   |
| `potent`    | Glowstone Block | `amplifier -> potent`   |

Every transition is generated only when both its source and target variants exist. Registered vanilla Regular, Extended (`long`), and Strong transitions remain in the reconstructed vanilla recipe set; missing vanilla-family and custom-effect tiers are generated under Overbrew.

### Container conversion

```json
{
  "splash": {
    "reagent": {
      "items": ["minecraft:gunpowder"]
    },
    "container_conversion": {
      "from": "regular",
      "to": "splash"
    }
  }
}
```

Both names must exist under `forms`.

Vanilla convertible potion IDs are derived from every configured base input and output plus every configured vanilla potion state. No separate conversion list is required.

### Global modifier

```json
{
  "no_particles": {
    "reagent": {
      "items": ["minecraft:resin_clump"]
    },
    "global": {
      "apply_to": ["custom", "vanilla"],
      "input_show_particles": true,
      "output_show_particles": false,
      "components": {
        "minecraft:lore": [
          {
            "translate": "overbrew.lore.no_particles",
            "fallback": "No Particles",
            "italic": false,
            "color": "#FC7812"
          }
        ]
      }
    }
  }
}
```

| Key                     | Behavior                                                                        |
| ----------------------- | ------------------------------------------------------------------------------- |
| `apply_to`              | Selects `custom`, `vanilla`, or both. Defaults to both.                         |
| `apply_to_forms`        | Limits the modifier to logical forms such as `regular`. Defaults to every form. |
| `input_show_particles`  | Particle state required on the input. Defaults to `true`.                       |
| `output_show_particles` | Particle state written to the output. Defaults to `false`.                      |
| `components`            | Adds, replaces, or removes output item components.                              |

Example:

```json
{
  "components": {
    "minecraft:custom_data": {
      "overbrew": {
        "particle_free": true
      }
    },
    "minecraft:enchantment_glint_override": true
  }
}
```

The component patch is applied to the generated potion stack. A component value of `null` removes that component from the output.

A global modifier generates direct recipes for matching custom and vanilla states. It also mirrors compatible custom transformations, vanilla recipes, and container conversions when the modified states remain distinguishable. Only forms allowed by `apply_to_forms` participate.

## Custom effects

```json
{
  "custom": {
    "effects": {
      "absorption": {
        "effect_id": "minecraft:absorption",
        "display": "Absorption",
        "translation_suffix": "absorption",
        "color": 2445989,
        "variants": {
          "profile": "standard",
          "states": {
            "base": {
              "duration": 3600,
              "amplifier": 0
            }
          }
        },
        "brew": {
          "awkward": {
            "items": ["minecraft:golden_apple"]
          }
        }
      }
    },
    "cross_effect_transforms": []
  }
}
```

| Key                   | Behavior                                                               |
| --------------------- | ---------------------------------------------------------------------- |
| Effect object key     | Stable effect ID used by recipe paths and tags                         |
| `effect_id`           | Minecraft status-effect ID                                             |
| `display`             | English fallback name                                                  |
| `translation_suffix`  | Final segment of the generated potion translation key                  |
| `color`               | Default decimal custom potion color                                    |
| `variants.profile`    | Optional reusable support, state, and scaling profile                  |
| `variants.durations`  | Optional local override for the inherited number of duration upgrades  |
| `variants.amplifiers` | Optional local override for the inherited number of amplifier upgrades |
| `variants.mix`        | Optional local override for whether both axes can combine              |
| `variants.scaling`    | Optional local override for inherited scaling rules                    |
| `variants.states`     | Explicit, registered, or state-specific values                         |
| `brew`                | Inputs and reagents that create the base state                         |

### Automatic variant grids

The resolved `durations` and `amplifiers` values are the number of non-base upgrades enabled on each axis. With the shipped three-entry axes, `2` enables `long` (Extended) and `prolonged`, or `strong` and `potent`. `mix` defaults to `true` and generates every supported combination. These fields normally come from the selected profile.

The Absorption example generates:

```text
base
long
prolonged
strong
potent
long_strong
long_potent
prolonged_strong
prolonged_potent
```

A state may use any of these forms:

```json
{
  "states": {
    "base": { "duration": 3600, "amplifier": 0 },
    "long": "minecraft:long_example",
    "strong": [
      { "duration": 400, "amplifier": 2 },
      { "duration": 400, "amplifier": 1 }
    ],
    "potent": {
      "potion": "minecraft:stronger_example",
      "color": 1234567,
      "effects": { "duration": 200, "amplifier": 2 }
    }
  }
}
```

- A namespaced string uses that registered potion type.
- A plain object supplies or overrides values for a generated single-effect state.
- An array supplies or overrides values for a generated multi-effect state in family effect order.
- An empty object explicitly keeps a state generated without overriding its calculated values.
- A wrapper object may combine `potion`, `color`, and `effects` when a registered state needs additional metadata.
- Missing states are calculated from the selected profile and available explicit anchors.
- Without a profile, `states.base` is required and the family must provide enough information to resolve every enabled axis.

### Effect instances

- `duration` is measured in ticks. There are 20 ticks per second.
- `amplifier` is zero-based. `0` is level I, `1` is level II, and `2` is level III.
- `ambient` defaults to `false`.
- `show_particles` defaults to `true`.
- `show_icon` defaults to `true`, but cannot be checked by brewing predicates.
- `hidden_effect` may contain another effect instance recursively, but cannot be checked by brewing predicates.
- A directly brewed custom effect must resolve a `base` variant, either locally or through its profile.

Example with every supported output field:

```json
{
  "duration": 3600,
  "amplifier": 0,
  "ambient": false,
  "show_particles": true,
  "show_icon": true,
  "hidden_effect": {
    "id": "minecraft:absorption",
    "duration": 1200,
    "amplifier": 0,
    "ambient": true,
    "show_particles": false,
    "show_icon": false
  }
}
```

Only the outer effect's `duration`, `amplifier`, `ambient`, and `show_particles` values can participate in a brewing predicate.

### Names and translations

For regular Absorption, the generated custom name resembles:

```json
{
  "translate": "item.minecraft.potion.effect.absorption",
  "fallback": "Potion of Absorption",
  "italic": false
}
```

`translation_suffix` controls the final key segment. The selected form controls `potion`, `splash_potion`, or `lingering_potion`. `display` and the form's `fallback_prefix` create the fallback name.

A resource pack may define the translation key, while the fallback remains readable without it.

### Custom cross-effect transformations

```json
{
  "cross_effect_transforms": [
    {
      "id": "corrupt_haste_to_mining_fatigue",
      "from_effect": "haste",
      "to_effect": "mining_fatigue",
      "variants": "matching",
      "modifier": "corrupting"
    }
  ]
}
```

- `id` is used in generated recipe paths.
- `modifier` references a top-level modifier and uses its reagent.
- `variants: "matching"` creates one recipe for every variant shared by both effects.
- `variants` may instead be an explicit list of shared names.
- When `variants` is omitted, `from_variant` and `to_variant` select one pair and default to `base`.
- When no modifier key is provided, the transform's `id` is used as the modifier ID.

This allows one corruption declaration to preserve every shared internal variant, including `base`, `long`, `prolonged`, `strong`, `potent`, and mixed tiers, without duplicating entries.

## Vanilla potion model

```json
{
  "vanilla": {
    "potions": {
      "swiftness": {
        "display": "Swiftness",
        "translation_suffix": "swiftness",
        "color": 3402751,
        "effects": [
          {
            "id": "minecraft:speed",
            "duration": 3600,
            "amplifier": 0
          }
        ],
        "brew": {
          "awkward": {
            "items": ["minecraft:sugar"]
          }
        },
        "variants": {
          "profile": "standard",
          "states": {
            "base": "minecraft:swiftness"
          }
        }
      }
    },
    "cross_effect_transforms": []
  }
}
```

Each object under `vanilla.potions` represents one potion family. A registered Regular potion type is placed in `variants.states.base`; separate family objects for `long_...` and `strong_...` potion IDs are not required.

| Key                  | Behavior                                                                  |
| -------------------- | ------------------------------------------------------------------------- |
| Family object key    | Stable base family ID used by generated paths                             |
| `display`            | English fallback effect name                                              |
| `translation_suffix` | Translation suffix shared by every state in the family                    |
| `color`              | Default color used by generated custom-effect states                      |
| `effects`            | Regular-state effect templates and IDs used by calculations               |
| `brew`               | Optional direct recipe that creates the registered Regular state          |
| `variants`           | Registered potion types, supported axes, profile, and explicit state data |

A registered state string is authoritative for matching and output. When a vanilla family provides only a registered Regular string, the generator infers the first duration and amplifier potion IDs from the configured axis names, such as `minecraft:long_swiftness` and `minecraft:strong_swiftness`. The profile calculates their effect values.

An explicit object or array suppresses that inference for the selected state and keeps it as a generated custom-effect state. This is used by vanilla families such as Infestation, Oozing, Weaving, and Wind Charged, whose registered Long potion types do not exist. A wrapper with `potion` is reserved for special registered metadata, such as Strong Turtle Master's different color.

Multi-effect families use profile or local state arrays, with per-effect scaling maps available when needed. The generator validates calculated registered `base`, `long`, and `strong` states against the same values previously represented by separate vanilla potion entries.

### Vanilla variant extensions

The generator preserves the existing vanilla brewing files byte-for-byte apart from their configured reagent tag routing. New Prolonged, Potent, mixed, and previously unsupported Extended variants are written as Overbrew-owned recipes under the custom namespace.

For Swiftness, the registered transitions remain vanilla recipes:

```text
Swiftness -> Long Swiftness
Swiftness -> Strong Swiftness
```

Overbrew adds only missing graph edges, such as:

```text
Long Swiftness -> Prolonged Swiftness
Strong Swiftness -> Potent Swiftness
Long Swiftness -> Long Strong Swiftness
Strong Swiftness -> Long Strong Swiftness
```

Generated vanilla-family states support regular, splash, and lingering conversions, particle-free chains, and matching corruption tiers.

### Vanilla cross-effect transformations

```json
{
  "cross_effect_transforms": [
    {
      "from": "leaping",
      "to": "slowness",
      "modifier": "corrupting"
    },
    {
      "from": "long_leaping",
      "to": "long_slowness",
      "modifier": "corrupting"
    }
  ]
}
```

Each configured vanilla transformation is reconstructed exactly as listed. When both referenced potion families define compatible variant grids, the generator also extends the transformation across additional matching generated tiers without duplicating the registered recipes already listed in `vanilla.cross_effect_transforms`.

For example, the configured Leaping-to-Slowness routes preserve their existing Regular, Extended, and Strong recipes while compatible Prolonged, Potent, and mixed variants are generated from the two families' shared variant names.

## Forms

```json
{
  "forms": {
    "regular": {
      "item": "minecraft:potion",
      "translation_item": "potion",
      "fallback_prefix": "Potion of "
    },
    "splash": {
      "item": "minecraft:splash_potion",
      "translation_item": "splash_potion",
      "fallback_prefix": "Splash Potion of "
    },
    "lingering": {
      "item": "minecraft:lingering_potion",
      "translation_item": "lingering_potion",
      "fallback_prefix": "Lingering Potion of "
    }
  }
}
```

- `item` is the Minecraft potion item ID.
- `translation_item` is used in generated translation keys.
- `fallback_prefix` is prepended to the effect's fallback display name.

Direct custom brews and same-form transformations are generated for every configured form. Container modifiers define transitions between forms.

## Configuration errors

Generation stops when the config is ambiguous or invalid, including:

- unknown forms or modifiers;
- empty reagent item lists;
- missing cross-effect states or requested matching variants;
- invalid variant axes, support limits, profiles, anchors, or calculated values;
- indistinguishable potion states;
- overlapping recipes or self-reapplying modifiers;
- overlapping generated tags when tag-overlap checks are enabled;
- invalid automatic adjustment strategies, modifier references, or potion-type IDs;
- automatic adjustment that cannot find a unique matchable state;
- conflicting settings for one tag path;
- generated tag cycles;
- unsupported `auto_generate` names;
- rules without a matcher or inferable `convert` path;
- disabled tags that resolve to more than one direct item;
- duplicate generated recipe paths;
- component-preserving tipped arrows enabled without a configured `lingering` form.

Fix the reported entry and run the generator again. Conflict errors include the matching routes and the most relevant settings or data changes.
